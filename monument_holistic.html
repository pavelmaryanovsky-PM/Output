📥
Phase 1 - Acquisition (Anonymous)
User has no account. No device bound. Journeys: emailExists → otpVerifyPhone → createMSParty
ANONYMOUS ▶
User does
Enters email
Receives & enters SMS OTP
Fills registration form (name, DOB, address, tax, account)
Confirms details on review screen
System does
SF lookup: does email exist as partyId?
Creates CSM token (15-min TTL) to chain steps
RiskID check on actionToken before each critical step
SF_CREATE_MEMBERSHIP_PARTY called with all form data
partyId returned - permanent identity established
Email verification link sent (NA106 notification)
Failure modes
Duplicate email → hard stop, shown info screen
RiskID DENY → silent hard stop
SF create party fails (non-200) → error info shown, reject
CSM mismatch (email/phone don't match token) → reject
OTP expires (2 min) → must restart OTP step
Key handoff: The CSM (cross-session message) is the spine of this phase. It's created in emailExists, consumed and re-issued in otpVerifyPhone (now carrying verified phone too), then consumed again in createMSParty. Each step can only proceed if it holds the correct token - this prevents step-skipping attacks. If any step fails, the whole chain must restart.
📱
Phase 2 - Device Binding
User exists. Device is new. Journey: default_bind (4 versions)
BIND ▶
User does
Opens app on new/fresh device
Approves on existing device (push) OR enters SMS OTP (if no old device)
Creates PIN on new device
Optionally sets up biometrics
System does
Jailbreak / root check first (hard gate)
Routes to correct version via filter_criteria params
SF lookup: get partyId for this user
SF_Get_party_details: fetch phone for OTP fallback
Adds device to device group ee356f85
Stores partyId in user custom_data
Failure modes
Jailbroken device → hard stop, reason: jail_broken_device
RiskID DENY (v: bind_withriskid) → hard stop
No bound devices + no old device → OTP fallback path
email_not_found in SF → reject
PIN lock (5 failures in 30min) → 30-min device lockout
Version routing logic: The bind journey has 4 active versions selected by filter_criteria on request params. bindViaPinAndPartyDetails = v8 MSR (skips all checks, just registers PIN + sets custom_data). RiskID param present = risk-gated bind. nextRelease=v7.0 = v7 flow. Default = original flow. The mobile app controls which version fires by what params it sends.
🔓
Phase 3 - Authentication (Login)
Device is bound. User returns. Journey: Login (2 versions: v3 disabled, login_lite active)
AUTH ▶
User does
Opens app
Biometrics or PIN (whichever is default)
Optionally: "Forgotten PIN" → SMS OTP → reset PIN
Optionally: memorable word check (if memorableWordSet=true)
System does
Loads user custom_data (emailVerified, marketingPreferenceSubmitted, etc.)
If partyId exists → SF_Get_party_details to refresh user state
SF_Memorable_Word_State_Check if partyId exists
Sends userData JSON to app (all flags for routing)
Enriches token with partyId claim
Registration check: device in group OR partyId not null
If applicationSubmitted=false → returns registrationData
Failure modes
Device not in group AND partyId null → redirect to Registration
emailVerified=false → app handles (not blocked at journey level)
5 wrong PINs → 30-min lockout
Forgotten PIN: OTP loop max 5 iterations
SF unreachable → partyId not enriched in token
login_lite vs v3: v3 is disabled. login_lite is active. The only real difference: login_lite hardcodes memorableWordSet: true in the userData JSON regardless of actual state. This means the memorable word check is bypassed at the journey level and pushed to the app to decide. Worth flagging as a potential discrepancy between journey truth and app truth.
⚙️
Phase 4 - Servicing (Post-Login)
User is authenticated. Journeys: Registration, CompleteIdv, update-phone, submit-application, store-applicationID, resend-email-verification
SERVICING ▶
Registration journey
Full onboarding form: name, DOB, address, tax, account
Review loop (editable, up to 30 iterations)
SF_CREATE_PARTY with all collected data
Adds device to group ee356f85 on completion
Token enriched with partyId
Servicing actions
CompleteIdv → sets completedIDV=true in custom_data
submit-application → applicationSubmitted=true, clears sensitive data fields
store-applicationID → saves applicationID to custom_data
update-phone → re-verifies new number via OTP, updates custom_data, sends NA104
resend-email-verification → re-triggers email verification flow
Registration-biometrics → registers biometrics post-login (session validate, no new token)
State written to custom_data
partyId (permanent)
applicationSubmitted (bool)
completedIDV (bool)
emailVerified (bool)
marketingPreferenceSubmitted (bool)
firstName, lastName refreshed from SF on each login
Sensitive fields (address, tax, DOB) nulled after submit
Data lifecycle: Sensitive PII (address, tax list, DOB, sort code, account number) is collected during Registration, stored in custom_data, submitted to SF, then explicitly nulled by submit-application. This is deliberate - Monument treats FlexID as a transit layer, not a PII store. SF is the system of record.
💳
Phase 5 - add-new-device (Ongoing)
Recurring. Any time user gets a new phone. Journey: add-new-device (3 versions)
LIFECYCLE ▶
Version 1 (default - disabled)
State: DISABLED
Registers biometrics first, then adds to device group
Fetches partyDetails + memorableWordState from SF
Sends NA103 (new device notification)
Version 2 (without biometrics)
State: ACTIVE
Filter: 6 specific Samsung/TCL device models only
Skips biometric registration entirely
Direct to device group add → SF data fetch → userData JSON
Hardcoded workaround for hardware incompatibility
Version 3 (v7 standard)
State: ACTIVE (all other devices)
Registers biometrics (with cancel escape)
Adds to device group
Fetches SF data, builds full userData JSON
Conditionally includes registrationData if applicationSubmitted=false
Sends NA103 notification
⚠️ Hardcoded device model list: The 6-model exclusion list (samsung/SM-G955F, SM-J330FN, SM-A105FN, Teclast/P80X_EEA, SM-A260F, TCL/5031G) is a maintenance liability. Every new incompatible device requires a journey deployment. This should be externalised to a config or device-group lookup.

Every external system this journey touches - and what breaks if it goes down.

☁️ Salesforce (SF)
SF_Get_party_id → email dedup check (acquisition + bind)
SF_Get_party_details → phone for OTP, name, flags (login, bind, forgotten pin)
SF_CREATE_PARTY → Registration final step
SF_CREATE_MEMBERSHIP_PARTY → Membership acquisition
SF_Memorable_Word_State_Check → login + add-new-device
SF_Get_Memorable_Word_Hint → memorable word auth
SF outage = Registration blocked, Login degraded (token has no partyId), Forgotten PIN broken
🤖 RiskID
Called via get_riskid_recommendation_userless(actionToken)
Used in: bind, CreateAccountDuplicatedEmail, emailExists, otpVerifyPhone, createMSParty
Returns: recommendation.type = ALLOW | DENY | (other → treated as ALLOW)
Service unavailable (catch) → returns synthetic DENY response
RiskID outage = entire acquisition flow blocked (fail-closed). Intentional design choice.
Logged to custom_activity_log: riskId_service_success / error / unavailable
📧 Notification Service
secure_notifications(code, params) → wraps all outbound comms
NA103 = new device registered notification
NA104 = phone number updated notification
NA106 = email verification (send_verification_email in membership flow)
NA107 = forgotten PIN reset notification
Notification failure is NOT blocking - flows proceed regardless. Silent failure risk.
📱 SMS / OTP Provider
Provider: otp-internal-hotp
Sender IDs: "monument" (prod), "monumentQA" (test/v7), "monumentPreProd" (membership flows)
6-digit code, 120-second expiry, wrong-inputs-only failure counting
3 different sender IDs across versions = potential user confusion on SMS origin
Phone number constructed as: mobileCountryCode + slice(mobileWithoutCode, 1, 10)
🔒 Device Group (ee356f85)
Single device group: "idvcheckpartyId-devicegroup"
Added on: Registration completion, add-new-device
Checked on: Login (devicesInDeviceGroup custom function)
Acts as the "trusted device" gate - login falls back to Registration if device not in group AND partyId null
Single device group for all users = no segmentation by product, risk tier, or geography
🧮 Custom Functions
Is_OS_Android() / Is_OS_iOS() → OS detection for version routing
devicesInDeviceGroup() → login device check
GenerateJWTWithPartyId() → membership flows, signs partyId into JWT
These are opaque - failure behaviour unknown from config alone

Pattern observation: Monument's versioning strategy is additive - old versions are disabled rather than deleted. This is good for rollback but creates config bloat over time. The filter_criteria expressions are the real feature flags - controlled entirely by what params the mobile app sends. This means the app and journey config must be kept in sync manually - there's no contract enforcement between them.

What this config tells us about Monument's product and engineering decisions - and what to watch.

🏗 Architecture
FlexID is a transit + auth layer only. SF owns all PII. Token = partyId handle.
CSM pattern (cross-session messages) is elegant for chaining anonymous flows securely without sessions.
Mobile app drives version selection via params - journey config is passive. Tight coupling.
Procedures (reusable sub-flows) used well: riskIdCheck, memorable_word, validate_mobile, dob_journey etc.
⚠️ Technical Debt
Hardcoded device models - needs externalisation to config or dynamic lookup.
login_lite always returns memorableWordSet=true - creates false state. Should either enforce or remove.
Dead code: jailbreak block in bind_withriskid evaluates "false == true" - never fires.
3 SMS sender IDs across versions suggests QA config bleeding into production paths.
Disabled versions accumulating - no cleanup cadence visible.
✅ What's done well
RiskID integration is thorough - covers acquisition, bind, membership flows. Fail-closed is the right call for a bank.
CSM chaining prevents step-skipping attacks cleanly.
PII cleared after submission (nulled in submit-application) - minimal data retention in FlexID.
Lockout rules are granular and per-authenticator - not a blunt instrument.
Biometric fallback to PIN with threshold (5 failures) is well thought out.
🛣 Product Observations
Two distinct onboarding funnels: Banking app (device-first) vs Membership savings (web/anonymous-first). Different trust models.
v7.0 / v8 versioning in params suggests aligned mobile app release cadence with journey config.
Memorable word is a UK banking compliance artefact (FCA guidance on step-up auth). Its "always true" state in login_lite suggests it's being soft-deprecated.
Recovery journey (policy_id: Recovery) exists but is disabled - account recovery not yet live or handled differently.
🚨 Immediate Actions
Fix login_lite: either restore SF memorable word check or remove the field from userData to stop misleading the app.
Fix dead jailbreak code in bind_withriskid (condition evaluates "false").
Externalise device model exclusion list from journey config.
Align SMS sender IDs - "monumentQA" should not appear in production-routed versions.
Document the app/journey param contract formally to prevent version misrouting.
🔭 Future Considerations
Recovery journey needs to be activated or explicitly decommissioned.
Device group segmentation (risk tier, product type) would enable more granular access policy.
SF as single dependency is a resilience risk - consider caching partyId/flags in FlexID custom_data with TTL.
Notification failure handling - at minimum log + alert; ideally retry or user-facing feedback.
Version cleanup cadence needed - disabled versions should be archived after 2 release cycles.
