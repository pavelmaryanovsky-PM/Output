Topic 1 - Strike management (FLORIE)
Cross-application failed authentication counting and reset
PCR-739 - Cross-app counting PCR-925 - Global reset API PCR-416 - Done (partial) PCR-355 - Done
Context
  • FlexID tracks failed logins per-app by default (FLORIE system)
  • "User" scope (cross-app counting) already exists but does not support per-authenticator thresholds
  • "Authenticator across devices" scope supports per-authenticator thresholds but is limited to one app
  • PNC needs both simultaneously - different thresholds AND cross-app counting
  • PNC has frozen new license purchases until resolved
Problem in one sentence

A user can accumulate strikes across PNC's apps without ever triggering a global lockout, and a successful auth on one app leaves strikes intact on all others.

Commercial impact
  • Blocking production rollout
  • License expansion frozen until resolved
  • Relationship health described as "very negative" in PCR
Product requirements
PCR-739 Cross-app, per-authenticator strike counting
"As a FlexID administrator, I want to configure failed authentication rules that count strikes per authenticator type across all applications, so that a user's full failure history is enforced regardless of which app they use."
Acceptance criteria
  • A new FLORIE counting scope is available: "Authenticator across all applications"
  • When this scope is selected, strike counts for each authenticator type (e.g. password, OTP) accumulate across all apps in the tenant
  • Different thresholds can be configured per authenticator type within this scope (e.g. 3 for password, 4 for OTP)
  • Existing scopes ("User", "Authenticator across devices") are unaffected
  • The new scope is configurable via the admin console FLORIE rule builder
Out of scope
  • Admin console UI changes beyond the new scope option in the rule builder
  • Cross-tenant strike counting
  • Automatic migration of existing FLORIE rules to the new scope
PCR-925 Global strike reset API
"As a developer integrating FlexID, I want an API endpoint to reset all strike counts for a user across all applications, so that a successful authentication in one app clears the user's full failure history."
Acceptance criteria
  • A new endpoint is available in the FlexID user management API: reset all strike counts for a given user across all applications
  • The endpoint accepts a user identifier and clears all FLORIE strike records for that user tenant-wide
  • The endpoint returns a success/failure response
  • No automatic invocation - the caller (PNC's backend) is responsible for triggering it on a successful auth event
  • The API is documented with explicit guidance that the caller is responsible for ensuring this does not conflict with their own FLORIE rule configurations
Out of scope
  • Admin console UI for triggering resets
  • Automatic reset logic built into FlexID flows
  • Product-side enforcement of cross-app FLORIE policy consistency
  • App-group scoped reset (may be considered in a future iteration)
Open before grooming - confirm with CX
  • Is PCR-925 the primary ask, superseding PCR-739, or are both required?
  • Should the new counting scope apply tenant-wide or be configurable per FLORIE rule?
  • Is audit logging required when a global reset is triggered?
  • Are these one engineering effort or two separate tickets?
Topic 2 - Kafka event streaming
Payload customization and schema registry support
PCR-773 had a delivery deadline of end of May 2026 for PNC's Retail Alpha Pilot. Confirm current status with CX before grooming.
PCR-773 - Payload customization (Blocker) PCR-865 - Schema registry (Blocker) PCR-819 - Schema registry (Citizens Bank)
Context
  • FlexID sends auth events to Kafka via Activity Log and Event Streaming connectors
  • Messages use StringSerializer - fixed schema, no customization
  • PNC's Kafka topics enforce a Schema Registry - FlexID messages fail deserialization
  • PCR-865 (PNC) and PCR-819 (Citizens Bank) describe the same gap independently
  • PCR-865 calls for 773 and 819 to be delivered together
Problem in one sentence

FlexID Kafka messages cannot be consumed by PNC's pipelines because they don't carry the required schema format, and their content cannot be shaped to match downstream system expectations.

Commercial impact
  • Blocking development for PNC's Retail rollout
  • Retail Alpha Pilot deadline was end of May 2026
  • Same gap affects Citizens Bank (PCR-819) - broader demand signal
Product requirements
PCR-773 Configurable Kafka event payload
"As a FlexID administrator, I want to control the content and structure of Kafka messages published by FlexID, so that downstream systems receive events in the shape they expect without requiring external transformation pipelines."
Acceptance criteria
  • Administrators can configure which fields are included or excluded from the Kafka message payload
  • Administrators can configure the message key (currently hardcoded to device_session_id)
  • Configuration is available for both Activity Log and Event Streaming Kafka connectors
  • Configuration is managed centrally via the admin console - not per-event scripting
  • Multiple schema versions per topic are supported
  • Existing default behavior (no customization) remains the default - opt-in only
Out of scope
  • Unrestricted custom scripting or code injection into event payloads
  • Real-time event enrichment from external systems at publish time
  • Retroactive transformation of already-published events
PCR-865 PCR-819 Schema Registry support for Kafka connectors
"As a FlexID administrator, I want to configure a Schema Registry for FlexID's Kafka connectors, so that messages are published in a format that can be consumed by Schema Registry-enforced Kafka topics."
Acceptance criteria
  • Kafka connector configuration in the admin console accepts a Schema Registry URL and credentials
  • FlexID supports at minimum KafkaAvroSerializer and KafkaJsonSchemaSerializer as serializer options for message values
  • Messages published with a configured Schema Registry include the correct wire format (Confluent 5-byte magic-byte prefix)
  • Schema Registry configuration is available for both Activity Log and Event Streaming connectors
  • If no Schema Registry is configured, existing StringSerializer behavior is preserved
Out of scope
  • Transmit-hosted or managed Schema Registry - customer provides their own
  • Automatic schema registration or schema evolution management
  • Support for non-Confluent Schema Registry implementations (to be confirmed with R&D)

  • Are 773 and 865 one engineering effort or two separate workstreams?
  • Which connector surfaces are in scope - Activity Log, Event Streaming, or both?
  • Can PCR-819 and PCR-865 be consolidated into a single FLEXID ticket?
  • What is the current status of PNC's Retail Alpha Pilot given the May deadline has passed?
